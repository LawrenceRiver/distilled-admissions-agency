# Research Lab Protocol

Use this when improving the skill itself or when running a rigorous admissions research project.

## Principles

- Treat every external repository, agency page, forum post, and advisor framework as source material, not something to copy.
- Extract the reusable pattern, rename it into this skill's terminology, and implement it in the local schema.
- Keep a source log for what influenced the design or a recommendation.
- Separate product design research from applicant admissions research.

## Pattern Extraction Table

For each source, record:

| Source | Bright spot | Risk | Our renamed pattern | Implementation |
|---|---|---|---|---|

Examples of renamed patterns:

- "program values research" -> `route-map`
- "application tracker" -> `decision ledger`
- "multi-agent peer review" -> `advisor council`
- "CV prerequisite gate" -> `profile passport gate`
- "school selection command" -> `target scope mode`

## Research Loop

1. Define the question: what part of the admissions workflow is weak?
2. Collect comparable tools, agencies, and public workflows.
3. Extract reusable patterns without copying text.
4. Add or revise one local workflow or reference.
5. Validate with a realistic applicant scenario.
6. Run an advisor-council review before calling the change stable.

## Stability Criteria

A change is stable only when:

- it improves a concrete user flow,
- it keeps applicant data out of the skill package,
- it preserves official-source priority,
- it makes output more decision-ready,
- it reduces rather than increases hallucination risk.
